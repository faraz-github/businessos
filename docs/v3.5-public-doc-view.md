# v3.5 — Public Document View Verification

Spec link: §8.

## Files reviewed

- `app/doc/[token]/page.tsx` (55 lines) — the server page that fetches
  the document by share_token and renders the PIN gate or the full
  view depending on access_code state.
- `app/doc/[token]/document-view.tsx` (1119 lines) — the full
  client-rendered document with print CSS.

## Verification checklist

### Layout — after PIN entry
- ✅ Header bar exists: business name + tagline + logo, doc-type badge.
- ✅ Print button (top-right of doc, in a `.no-print` wrapper that
  hides when actually printing) — single visible action,
  unambiguous.
- ✅ Sign action: the signing flow is rendered for contracts /
  proposals as appropriate; signed state replaces it.
- ✅ Returns to a "signed on {date}" indicator after signing.
- ✅ Mobile: the document scales correctly because the A4 wrapper has
  `max-width: 100%` in print and shrinks reasonably on small viewports
  (the inner content is a column layout, not a fixed-width grid).

### Print CSS audit (lines 484-540 of document-view.tsx)

Posture is correct:

- `body, html { background: #fff; margin: 0; padding: 0 }` — strips
  the page-bg wrapper.
- `* { -webkit-print-color-adjust: exact }` — preserves brand colors.
- `.no-print { display: none }` — hides the print button itself.
- `body > div { background: #fff; padding: 0; min-height: unset }` —
  collapses the outer wrapper.
- `.doc-page { width: 100%; max-width: 100%; margin: 0; padding: 0;
  border-radius: 0; box-shadow: none; overflow: visible }` — fills
  the page edge-to-edge.
- `@page { size: A4 portrait; margin: 12mm 14mm }` — proper page setup.
- Page-break controls inside section blocks (line 510+).

### After-PIN flow

The page has two render branches based on whether `access_code` was
verified. Verified branch renders `<DocumentView>` immediately, no
loading flash. Unverified branch renders the PIN entry form. There's
no intermediate loading state to flash.

### What I CHANGED in step 8

- Added `fields.paid_date` rendering to the public invoice view. Now
  shows "Paid: {date}" in green next to the existing "Due: {date}"
  line. Was previously only readable from the dashboard. Critical for
  v3.5 since step 4 introduced `paid_date` as the canonical "when was
  this actually paid" field.

### What I did NOT change

- Header layout, business-name rendering, logo treatment — all fine
  as-is.
- Print CSS — comprehensive, no holes I could find.
- Sign / signed-state UX — works correctly.
- Download PDF button — there isn't one, deliberate. The "Save as
  PDF" button uses the browser print dialog, which is the right
  trade-off (no @react-pdf rendering pipeline to maintain server-side
  for public viewing). The button label is honest about what it does.

## Cross-deps

- Step 4 added `paid_date` to invoice fields. Now displayed publicly.
- Step 6's signature section pinning has no impact on the public view
  — `active_sections_*` arrays are read in order whichever way they're
  serialized.
- Step 7's required-field validation prevents incomplete docs from
  being shareable in the first place — the public view never has to
  defensively render against a half-baked doc, because shareDocument
  now refuses to mint a token for one.

## Suggested manual smoke test

1. Create a draft invoice in paperwork → mark fields incomplete →
   try to Send → should fail with "required fields missing" toast.
2. Fill in all required fields → Send → access_code shown.
3. Open `/doc/[token]` in incognito → enter access_code → see the doc
   with header, print button top-right, full content.
4. On the invoice doc, check that "Paid: {date}" line appears below
   "Due: {date}" if you've marked it paid via the finance page.
5. Click "Save as PDF" → browser print dialog opens → preview shows
   only document content (no print button, no page wrapper bg).
6. Test on mobile viewport (devtools): doc still readable, print
   button accessible.
