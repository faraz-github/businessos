- **[Major]** ŌĆö measurable bug, inconsistency, or cross-step dependency

## Findings ŌĆö by severity

### [Critical-1] `get_home_stats` RPC is never called ŌĆö 8 queries per home load when 1 would do

**File:** `app/dashboard/actions/home.ts:142-274` (current implementation).
**Spec link:** performance contract ("no client-side waterfalls") + the
migration 017 header comment that sold the RPC as a ~8├Ś improvement.

**What I found:** Migration 017 added `public.get_home_stats(p_user_id, p_mode)`
ŌĆö a single PL/pgSQL function that returns the full dashboard payload as
JSONB in one round-trip. The code comment explicitly says it exists to
"cut dashboard load time from 8 sequential round-trips to 1."

Nothing in the codebase calls it. `grep -rn "\.rpc"` returns zero hits.
Both home pages call `getHomeStats` from `home.ts`, which still does the
8 sequential queries in JS.

**Consequence:** every dashboard visit pays the full 8-query latency tax.
On a cold connection to Supabase (IST ŌåÆ AWS Mumbai is ~40 ms RTT), that's
~320 ms of sequential waits before the home page paints ŌĆö and that's
per-mode, so switching from Personal to Agency reloads the whole thing.
The RPC cuts this to ~40 ms.

**Fix:** Replace the body of `getHomeStats(mode)` in `home.ts` with a
single `supabase.rpc('get_home_stats', { p_user_id: ownerId, p_mode: mode })`
call. The RPC already returns the exact shape the `HomeStats` interface
in `app/dashboard/personal/home/client.tsx:31-36` consumes. No client
code changes required.

### [Critical-2] Revenue is double-counted when both auto-log and manual payment logs coexist

**File:** `app/dashboard/actions/transactions.ts:134-197` (`markInvoicePaidAction`)
+ `app/dashboard/personal/finance/page.tsx:781-811` (`AddTransactionModal`).
**Spec link:** ┬¦3 "Invoice Ōåö transaction linking ŌĆö when `markInvoicePaidAction`
logs the income transaction, is it linked back to the invoice (for dedup)?"

**What I found:** `markInvoicePaidAction` inserts an income transaction
every time an invoice is marked paid (transactions.ts:172-182). The
insert doesn't set `invoice_id` (schema has the column, action ignores
it). If the user also logs the same payment manually via
`AddTransactionModal` ŌĆö which the spec describes as a normal flow for
payments received before the invoice was created in the system ŌĆö the
same revenue appears twice in:

- `revenueThisMonth` (monthly income sum)
- The 5-month sparkline
- The transactions list (visible double entry)
- The `getHomeStats` money.revenueThisMonth the home page shows

**Fix:** 
1. In `markInvoicePaidAction`, include `invoice_id: parsed.data.id` in the
   transaction insert payload. The `transactions.invoice_id` column has no
   FK constraint (invoices-as-table was dropped in migration 012) so this
   is just data.
2. In `AddTransactionModal`, when the user is logging an income
   transaction, optionally let them link it to an unpaid invoice. If
   they do, the mark-paid flow reads that link and skips the auto-insert.
3. Alternative fix (simpler, less user-visible): in
   `AddTransactionModal`, if the user manually logs income with a
   description matching "Invoice {N}" of an already-paid invoice, show a
   warning "This invoice was already auto-logged on {date} ŌĆö log anyway?"
   Leaves the double-entry possible but prevents accidental dupes.

Recommending (1) + (2) because it's the principled fix and makes the
future "payment received partially" case tractable. (3) alone is
bandage-y.

### [Critical-3] Overdue writeback is client-only and fires from a useEffect-shaped callback

**File:** `app/dashboard/personal/finance/page.tsx:115-161`.
**Spec link:** ┬¦3 "is it firing too aggressively (on current date being
past due_date regardless of payment) or correctly (only for unpaid)?"

**What I found:** When the finance page loads, `loadData` walks the
invoices list in JS, detects rows whose `fields.due_date < today` and
DB status is not already `overdue`, then fires a single
`supabase.from('documents').update({ status: 'overdue' })` with an
`in('id', staleOverdueIds)` filter.

The logic **is** correctly scoped ŌĆö only non-paid, non-signed, past-due
rows update. That part is fine. The **architecture** is not:
- Anyone who doesn't open `/dashboard/personal/finance` never triggers
  the writeback. The home attention feed (home.ts:67-76) and home stats
  RPC (migration 017:96-111) both compute "overdue" on-the-fly by
  comparing `fields.due_date < today`, so the UI stays correct ŌĆö but the
  DB `status` column drifts from reality. Anywhere else that filters on
  `status = 'overdue'` (e.g. the finance page's own `invFilter === 'overdue'`
  branch on page.tsx:302) shows the wrong thing until someone visits the
  finance page.
- The writeback happens via the browser anon client. Under v3.5's proper
  RLS (step 1), that's fine because the `.eq('user_id')` + `.eq('mode')`
  filters match the RLS predicate. But this is coincidental safety ŌĆö the
  update could just as easily have been written without those filters,
  and no server-side guard would have caught it.

**Fix:** Remove the client-side writeback entirely. Every overdue-detection
call site derives overdue from `status not in ('paid','signed') and
(fields->>'due_date')::date < today` at query time:

- Home attention feed (`home.ts:67-76`) ŌĆö already does this Ō£ō
- Home stats RPC (`003_functions_and_seed.sql:96-111`) ŌĆö already does
  this Ō£ō
- Finance page (`finance/page.tsx:120-161`) ŌĆö computes overdue locally
  Ō£ō, then **also writes back**, which is the redundancy to remove.

Action: delete the `staleOverdueIds` logic from `loadData` (finance
page). The derived check on line 126-129 stays ŌĆö it sets the local
`status` string used for UI rendering, which is correct. Just drop the
lines 115-119, 152-161 that do the server write.

Keeping the `overdue` value in the `DocumentStatus` type and check
constraint ŌĆö they're still useful as a physical cache when set (e.g. a
future batch job could write them for faster filtering), and removing
them would ripple through a dozen UI sites that switch on `status`.
Treating the physical column as best-effort cache, derivation as
source-of-truth.

Cross-dep with step 4: once `paid_date` lands, add `and (fields->>'paid_date')
is null` to the three derivation sites (covered by Major-4).

### [Major-1] `monthlyBurn` doesn't normalize cycle ŌĆö will break silently when step 5 adds quarterly/semi-annual

**File:** `app/dashboard/personal/finance/page.tsx:180-181`.
**Spec link:** ┬¦5 (subscription cycles) + ┬¦3 (audit pass).

**What I found:**
```ts
const monthlyBurn = subscriptions.filter(s => s.status === 'active')
  .reduce((sum, s) => sum + (s.billing_cycle === 'annual' ? Number(s.cost) / 12 : Number(s.cost)), 0);
```

Branches only on `annual` vs everything-else. Once step 5 adds
`quarterly` and `semi_annual`, this treats a Ōé╣9000/quarter subscription
as Ōé╣9000/month ŌĆö 3├Ś the real monthly cost. Nothing typechecks against
it because `billing_cycle` is typed as the union and both new values
fall into the `else` branch.

**Fix:** Extract the normalization into a helper in `lib/utils/index.ts`
(per spec ┬¦5e):
```ts
export function monthlyEquivalent(cost: number, cycle: BillingCycle): number {
  const divisor: Record<BillingCycle, number> = {
    monthly: 1, quarterly: 3, semi_annual: 6, annual: 12,
  };
  return cost / divisor[cycle];
}
```

Using a `Record<BillingCycle, number>` means adding a new cycle to the
type will fail to compile here until the helper is updated. That's the
whole point ŌĆö we want this to be a compile error next time, not a
runtime wrong answer.

Apply at every burn computation. Based on a grep, that's just the one
site in finance page. The home stats don't surface subscription burn.

### [Major-2] Income category hardcoded to `project_payment` on auto-log

**File:** `app/dashboard/actions/transactions.ts:178`.

**What I found:** `markInvoicePaidAction` always inserts with
`category: 'project_payment'`. The UI (`INCOME_CATS` at
finance/page.tsx:33-39) offers `initial_payment`, `final_payment`,
`retainer`, `other_income` as alternatives. None of these are
selectable at mark-paid time, and the server action doesn't accept a
category parameter.

Consequence: a user who's clearly invoicing for (say) a retainer or
final payment has their transaction logged as `project_payment`. The
category breakdown on finance page is therefore always biased toward
`project_payment`.

**Fix:** Add an optional `category` parameter to the
`markInvoicePaidInputSchema`, defaulting to `'project_payment'`. Add a
small dropdown to the mark-paid confirmation flow on the finance page
(or let step 4, which is adding a mark-paid modal for `paid_date`,
include the category picker there too). Default to `project_payment` so
existing behavior is preserved.

Flagging as Major because it shapes real financial data, but not
Critical because the amounts are correct ŌĆö only the bucketing is wrong.

### [Major-3] Outstanding calculation bug ŌĆö missing `overdue` in the "outstanding" filter

**File:** `app/dashboard/personal/finance/page.tsx:183`.

**What I found:**
```ts
const outstandingTotal = invoices.filter(i =>
  ['sent','viewed','overdue'].includes(i.status)
).reduce(...);
```

This is actually correct. BUT on `home.ts:170-173` (the home stats JS
version), `outstandingTotal` includes any invoice where
`status not in ('paid','signed')` ŌĆö that's a superset: `draft` rows get
counted as outstanding on the home page but not on the finance page.
The RPC version (migration 017:110-111) matches `home.ts`, not
`finance/page.tsx`.

So an invoice in `draft` status counts as "outstanding" on Home but not
on Finance. The mismatch breaks user trust ("home says Ōé╣50k
outstanding, finance shows Ōé╣30k ŌĆö which is right?").

**Fix:** Decide a canonical definition and enforce it everywhere:
- **Definition A (stricter):** outstanding = sent, viewed, or overdue.
  Drafts aren't outstanding because they haven't been sent to a client
  yet ŌĆö the client doesn't owe anything.
- **Definition B (looser):** outstanding = everything unpaid. Drafts
  included because they represent money the owner expects to invoice.

Recommending **A**. A draft isn't a debt. Matches the finance page
already; the change is in `home.ts` and the migration 017 RPC body.

Cross-dep with Critical-1: the fix for Critical-1 (switch home.ts to
the RPC) and this fix (align the definition) happen in the same commit.
I'll update the RPC in `003_functions_and_seed.sql` alongside the code
change.

### [Major-4] Overdue / paid detection needs belt-and-braces for step 4 `paid_date`

**Files:** `app/dashboard/actions/home.ts:67-76` (attention feed),
`app/dashboard/actions/home.ts:174-179` (JS stats),
`supabase/migrations-consolidated/003_functions_and_seed.sql:96-111`
(RPC).
**Spec link:** ┬¦4 testing case 4: "Overdue detection should ignore any
invoice with a `paid_date` set (not just `paid` status, belt-and-braces)."

**What I found:** All three overdue-detection call sites filter on
`status not in ('paid','signed')`. Step 4 adds `paid_date` inside the
invoice's `fields` JSON. After step 4 lands, a malformed / edge-case
invoice whose `status` stayed `sent` despite a `paid_date` being set
would incorrectly appear as overdue.

**Fix:** In every overdue-detection site, add the additional guard:
`and (fields->>'paid_date') is null` (SQL) or
`!f?.paid_date` (JS).

This is a step-4 concern but flagging here so the audit covers it and
the step-4 implementation can check these three sites off.

### [Major-6] `home.ts` uses the old auth pattern ŌĆö five fixable sites

**File:** `app/dashboard/actions/home.ts:10,144,278,295,312`.

**What I found:** Every action in `home.ts` uses:
```ts
const session = await getSession();
const user = session ? { id: getOwnerId(session) } : null;
if (!user) return [];
```

The canonical pattern (subscriptions.ts) is:
```ts
const session = await requireSession();
const ownerId = getOwnerId(session);
```

The current pattern silently returns `[]` for unauthenticated requests ŌĆö
that shouldn't be possible because `proxy.ts` blocks them, but the
silent fallback means a real bug (e.g. a cookie rotation issue) surfaces
as empty data instead of a clear 401. Minor functional impact but worth
fixing while we're in this file for Critical-1 and Major-3.

**Fix:** Convert every action in `home.ts` to the canonical pattern.
Since I'm touching `getHomeStats` anyway for Critical-1, doing the other
four sites is a 10-minute rewrite.

### [Minor-2] `formatINR` default rounds rupees, loses paisa precision

**File:** `lib/utils/index.ts:60-76`.

**What I found:** Default call path uses `minimumFractionDigits: 0,
maximumFractionDigits: 0`. Call sites can opt into decimals with

**Fix:** None, documenting. If step 7's paperwork audit finds that
invoice PDFs need paisa precision for compliance, the fix is
`formatINR(amount, true)` at those call sites.

### [Minor-3] `calculateGST` rounds before totaling

**File:** `lib/utils/index.ts:86-89`.

**What I found:**
```ts
export function calculateGST(amount: number, rate: number = 18) {
  const gst = Math.round((amount * rate) / 100);
  return { gst, total: amount + gst };
}
```

For `amount=12345, rate=18`: `gst = Math.round(2222.1) = 2222`, `total = 14567`.
Direct compute would be `12345 * 1.18 = 14567.1`. One rupee low.

For real businesses at small invoice volumes this is harmless ŌĆö GST
returns are rupee-rounded per line item anyway. Flagging because the
pattern is unintuitive.

**Fix:** Document the rounding behavior in a JSDoc. No code change.
