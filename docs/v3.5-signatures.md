# v3.5.1 — Signatures Migration Audit

Spec link: internal follow-up to v3.5. Two related changes bundled:

1. **Client signatures stop being stored as base64.** The public
   signing flow now uploads drawn signatures to the `document-media`
   bucket and stores the storage path in `signatures.signature_data`
   and `documents.fields.client_signature.data`.
2. **Saved signatures on brand profile.** Personal and agency brand
   profiles gain an optional reusable signature (drawn in app or
   uploaded as a stamp), surfaced as a PICKER option in the paperwork
   editor alongside Type / Draw.

These two landed together because they share `SignaturePad` — a new
shared canvas component that replaced three near-identical
implementations.

---

## Problem — the base64 side

The v3.5.0 codebase had the project rule "never store images in the
database as `data:image/png;base64,…`". That rule was obeyed for
logos (which went to `brand-assets` via `lib/storage/upload.ts`) but
violated in three places for signatures:

1. **Public signing modal** (`app/doc/[token]/document-view.tsx`).
   The canvas called `toDataURL('image/png')` and wrote the resulting
   data URL straight to `signatures.signature_data` and into
   `documents.fields.client_signature.data`. The base64 blob was
   sometimes 40–60 KB per signature; bloats document payloads.
2. **Creator signing in paperwork editor** (`SenderSignatureField`
   in `app/dashboard/personal/paperwork/page.tsx`). Same pattern —
   `toDataURL` → `fields.creator_signature.data` → DB via autosave.
3. **No saved signature story at all.** Every outgoing document
   required the creator to re-sign from scratch. There was no place
   to save a signature to a brand profile, so users reached for the
   workaround of typing their name every time (losing the visual
   artefact of a drawn signature).

## Policy

| Signature source | Storage | Location |
|---|---|---|
| Typed (client or creator) | Plain text in `signature_data` / `fields.*_signature.data` | DB only — no storage |
| Drawn (client, via `/doc/[token]`) | PNG bytes | `document-media/{ownerId}/{docId}/signatures/client-{nanoid}.png` — private, signed URLs per render |
| Drawn (creator, via paperwork editor) | PNG bytes | `document-media/{ownerId}/{docId}/signatures/creator-{nanoid}.png` — same private posture |
| Saved (brand_profile `signature_url`) | PNG/JPG/WebP | `brand-assets/{ownerId}/{mode}/signature-{timestamp}.{ext}` — public URL, same posture as the logo |

What gets stored in the DB after this change:

- `signatures.signature_data` — either the typed name (typed signatures) or the storage path (drawn). Never a data URL.
- `documents.fields.client_signature.data` — same shape plus the brand-asset HTTPS URL for saved-signature picks.
- `documents.fields.creator_signature.data` — same.
- `brand_profiles.signature_url` — brand-asset public HTTPS URL.

## Implementation

### Feature A — client signing via `/api/doc/sign`

New public POST route. Mirrors the `/api/doc/verify-code` pattern:

1. Look up the document by `share_token`.
2. Re-verify the access code submitted alongside the signature.
   Constant-time compare; expired codes return a specific error.
3. Refuse if the doc isn't in a sign-able status (`sent` or `viewed`).
   `signed`/`paid` returns 409, not 401, to distinguish user-error
   from a replay attack.
4. For drawn signatures: decode the base64 PNG data URL the browser
   sent, upload the bytes to `document-media` at
   `{ownerId}/{docId}/signatures/client-{nanoid}.png`, store the
   STORAGE PATH (not URL) in `signatures.signature_data` and in
   `documents.fields.client_signature.data`.
5. For typed signatures: store the typed string as-is.
6. Return the `client_signature` shape for the browser to optimistically
   render.

Why upload server-side instead of directly from the browser: the
`document-media` RLS only allows inserts from authenticated users
scoped to their own folder. The signing caller is anonymous (share
link + access code). So the upload goes through this route, which
uses the service-role client and re-verifies the code before touching
storage.

### Public document view — signed URL resolution

`app/doc/[token]/page.tsx` (server component) now calls
`resolveSignatureUrls()` before passing the document to the client.
It walks `fields.client_signature` and `fields.creator_signature`,
detects storage paths (versus legacy data URLs or public HTTPS URLs),
and mints a fresh signed URL via `signedUrlFor()` for each. Expiry is
the default 1 hour — plenty for a single page view.

Backwards compatibility: if `data` starts with `data:` (legacy v3.4
base64) or `http(s)://` (saved-signature brand-asset URL), it passes
through unchanged. Only bare paths get signed.

### Feature B — saved signature on brand profile

Adds two columns to `brand_profiles`:

```sql
alter table public.brand_profiles
  add column signature_url  text,
  add column signature_type text check (signature_type in ('drawn', 'uploaded'));
```

Same bucket as logos (`brand-assets`), same path-builder
(`brandSignaturePath(ownerId, mode, mimeType)` ⇒
`{ownerId}/{mode}/signature-{timestamp}.{ext}`), same compression
profile (`'logo'` — ≤200KB / 512px long edge; signatures print at
postage-stamp size so this is generous).

Two new server actions in `app/dashboard/actions/brand.ts`:

- `uploadBrandSignature(mode, formData, kind)` — `kind` is `'drawn'`
  or `'uploaded'`; persists on `signature_type`. Mirrors the logo
  upload 1:1 including delete-after-save for old paths.
- `removeBrandSignature(mode)` — clears both columns; best-effort
  storage delete.

Settings page grows a **Signature** card between Logo and Identity
with three tabs:

- **Saved** — shows the current saved signature with a Remove
  button. Only selectable when one is saved.
- **Draw** — uses the shared `SignaturePad` component. Save button
  exports the canvas as a PNG `Blob` directly (not a data URL) via
  `canvas.toBlob` and uploads via FormData.
- **Upload** — file picker with `compressImage(file, 'logo')` reused
  from the logo flow.

### Paperwork editor — `SenderSignatureField`

Rewritten to offer three tabs: **Saved · Type · Draw**. The Saved
tab is hidden (not disabled) when `brand.signature_url` is unset —
an empty-state tab adds noise without adding capability.

**Critical product decision**: the saved signature is a PICKER option,
never auto-applied. The user explicitly clicks "Use this" each time.
Rationale: some documents (e.g., internal amendments, one-off
proposals) shouldn't carry the same signature as formal outgoing
contracts; forcing a conscious pick preserves that choice. The
original request was unambiguous on this — see the consultation thread
that led to this work.

On Save, the three paths branch:

- **Saved**: `onChange({ type: brand.signature_type, data: brand.signature_url, date, name: brand.business_name })`.
- **Typed**: `onChange({ type: 'typed', data: name, date, name })`.
- **Drawn**: canvas → PNG Blob → FormData → `uploadCreatorSignature(docId, formData)` → storage path → `onChange({ type: 'drawn', data: path, date, name: '(drawn)' })`.

`uploadCreatorSignature` is a new authenticated action in
`app/dashboard/actions/documents.ts`. Ownership-checked against
`documents.user_id` before touching storage. Uses
`signaturePath(ownerId, docId, 'creator', 'image/png')` — same helper
as the public signing route, different `role` segment.

### The shared `SignaturePad` component

Extracted from three near-identical implementations in:

- `app/doc/[token]/document-view.tsx` (public signing modal)
- `app/dashboard/personal/paperwork/page.tsx` (SenderSignatureField)
- `app/dashboard/personal/settings/page.tsx` (brand signature card)

The component uses `forwardRef` + `useImperativeHandle` to expose
`getDataUrl()`, `getPngBlob()`, `clear()`, and `hasStrokes()`.
Caller-controlled export timing means we don't re-render on every
stroke just to keep a string in state — on mid-range mobiles that
lag was visible.

Note: mouse events only, no touch. Matches v3.5 behaviour. Touch
support via pointer events is a focused follow-up.

## Cross-deps

- **v3.5 step 2** (`document-media` bucket): this is the first UI
  consumer of that bucket. The "empty infrastructure" note in the
  v3.5 release notes is no longer accurate — update.
- **v3.5 step 4** (`paid_date`, invoice lifecycle): no interaction.
  Invoices aren't "signed" in the e-signature sense.
- **v3.5 step 7** (required-field validation): no change. The
  required schemas don't validate signature presence — signatures
  are a post-final action (client signs after receiving), not a
  pre-send requirement.
- **v3.5 step 8** (public doc view): the v3.5.0 verification
  checklist's "after-PIN flow" still holds; signed URL resolution
  happens transparently server-side.

## Migration for existing v3.5.0 data

If any v3.5.0 instance has already accumulated base64 signatures (the
bug this fixes), those rows stay readable:

- `resolveSignatureUrls()` in `app/doc/[token]/page.tsx` only signs
  entries that look like storage paths. Anything starting with
  `data:` passes through unchanged and renders the same as before.
- New signatures after upgrade go to storage cleanly.
- A one-off backfill script to move existing base64 blobs into
  `document-media` and rewrite the `fields` is straightforward but
  NOT shipped with this release — base64 data URLs are not a
  correctness bug post-patch, just a bloat issue. Write one only
  if storage sizes are a concern.

Schema migration itself is idempotent-friendly:

```bash
# Apply just the new incremental file to an existing v3.5.0 instance:
# supabase/migrations-incremental/021_brand_signatures.sql
```

For fresh deploys, the consolidated baseline (`001_schema.sql`)
already includes the new `signature_url` / `signature_type` columns.

## Test plan (manual)

1. **Client drawn signature end-to-end:**
   - Create a contract, share it with an access code.
   - Open `/doc/[token]` in incognito, enter the code.
   - Sign via the "Draw" tab → confirm the document now shows the
     signature.
   - In Supabase Storage, confirm a new file at
     `document-media/{ownerId}/{docId}/signatures/client-*.png`.
   - In the DB, confirm `signatures.signature_data` is a path
     (NOT `data:image/png...`).
2. **Client typed signature:**
   - Same flow, "Type" tab instead. Confirm no storage file created.
   - Confirm `signatures.signature_data` is the plain typed name.
3. **Saved signature roundtrip:**
   - Settings → Signature card → Draw → save.
   - Confirm a new file at
     `brand-assets/{ownerId}/personal/signature-{ts}.png`.
   - In the paperwork editor for a contract, open the sender
     signature picker — confirm "Use saved" tab appears.
   - Click "Use this" → confirm `fields.creator_signature.data` is
     the HTTPS URL of the saved signature (NOT a path, NOT base64).
4. **Upload stamp:**
   - Settings → Signature → Upload a PNG → save. Same as above.
5. **Saved signature is picker-only:**
   - With a saved signature present, create a new contract. Do NOT
     open the sender signature picker. Send the doc.
   - Confirm `fields.creator_signature` is still null — the saved
     signature was NOT auto-applied.
6. **Backwards compat:**
   - If you have a v3.5.0 backup with base64 signature data,
     restoring it + opening the public view should still render the
     signature (via the legacy-data-URL passthrough branch).
