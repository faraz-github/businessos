# v3.5 — Paperwork Validation Audit

> **Revised in v3.5.1** — see the "Revisions in v3.5.1" section at the
> bottom. The required schemas documented here were pared back, Zod
> paths no longer leak into user-facing errors, and a type coercion
> bug in `payment_schedule.amount` was fixed at the writer.

Spec link: §7.

## Problem

Some validations on the paperwork editor block the Save button when
sections aren't filled or fields are empty, even though autosave runs
and the document is genuinely a draft. The distinction between
"draft — anything goes" and "final — must validate" is muddled.

## Policy

| Status | Validations |
|---|---|
| `draft` | None. Save always succeeds. Autosave runs normally. |
| `final` | All required fields per doc type must be present and valid. Block transition to `final` if not. |
| `sent` | Same as `final`. Once sent, structure is locked for editing except via explicit "edit" actions. |
| `signed` / `paid` | Same as `final`. No structure changes allowed. |

## Required fields per doc type

Documented as Zod schemas in `types/schemas.ts` under
`<doctype>RequiredSchema`. These are stricter than the existing
`<doctype>FieldsSchema` (which allows partials for autosave).

### Proposal — `proposalRequiredSchema`
Required: `client_name`, `project_title`, `overview`, `timeline`,
`investment_amount` (> 0).

### Contract — `contractRequiredSchema`
Required: `parties.client`, `parties.freelancer`, `project_description`,
`payment_schedule` (at least one entry with non-empty trigger and amount > 0),
`governing_law`.

### Scope of Work — `sowRequiredSchema`
Required: `objectives`, `deliverables` (at least one),
`acceptance_criteria`.

### Requirements — `requirementsRequiredSchema`
Required: `functional_requirements` (at least one), `deadline`.

### Invoice — `invoiceRequiredSchema`
Required: `invoice_number`, `client_name`, `line_items` (at least one
with description, quantity > 0, rate > 0), `due_date`, `total` > 0.

### Delivery — `deliveryRequiredSchema`
Required: `project_summary`, `deliverables` (at least one).

## Implementation

1. **Server action**: `updateDocument` accepts an optional
   `target_status` parameter. If `target_status` is `'draft'` or
   omitted, no per-field validation. If `target_status` is `'final'`
   or above, the appropriate `<doctype>RequiredSchema.safeParse` runs
   on `fields`. If it fails, return `{ ok: false, error: <issues> }`
   without touching the DB.
2. **Page**: two save controls.
   - **Save as Draft** (the existing Save button, relabelled): always
     works, never validates field shapes.
   - **Mark as Final** (new button next to it): calls `updateDocument`
     with `target_status: 'final'`, surfaces validation errors as a
     toast or inline message.
   - The existing "Send" flow (`shareDocument`) implicitly wants
     final-quality data — it should run the same required check before
     generating share_token + access_code.

## What landed in step 7

- This doc.
- Per-doctype `<doctype>RequiredSchema` exports in `types/schemas.ts`.
- `updateDocument` server action accepts `target_status` and runs
  required validation when it's `final` or above.
- `shareDocument` server action runs the same required validation
  before generating the share token (an unsendable doc should never
  get a share URL).

## What did NOT land in step 7

- The "Mark as Final" UX button. The save handler in the paperwork
  editor still calls `updateDocument` with the document's existing
  status. Adding the dual-button UX is one focused PR — reviewing the
  paperwork editor's save UX holistically is worth more than tucking
  it into a v3.5 sub-step.
- Per-field error rendering inline (red borders, error text below
  fields). The current toast surfacing is fine for v3.5.

## Cross-deps

- Step 4 added `paid_date` to the invoice fields. The
  `invoiceRequiredSchema` does NOT require `paid_date` — paid_date is
  set by the mark-paid flow, not by editing the invoice in paperwork.
- Step 6 pinned signature sections to the bottom of the active
  sections list. The required schemas don't validate `active_sections`
  arrays — the bottom-pinning is structural, not data validation.

## Revisions in v3.5.1

Three related issues surfaced after v3.5.0 shipped:

1. **Validation messages leaked Zod internals.** A user who clicked
   Send on a contract with a string-valued amount saw:
   > Cannot send — required fields missing: payment_schedule.0.amount:
   > Expected number, received string; payment_schedule.1.amount: ...

   That is: the Zod path, the TypeScript type name, and a semicolon-
   separated list of every issue. Developer-facing text in a user-
   facing toast.

2. **The amount was stored as a string in the first place.** The
   payment-schedule add button pushed `newPayment` (the raw state
   where `amount` is the input element's string value) directly into
   `fields.payment_schedule`. The UI's total row already called
   `Number(p.amount)` defensively — so this was invisible in the
   editor but broke validation when a type-strict check ran. Invoice
   line-items did this correctly (coercion in `addItem()`);
   payment-schedule did not.

3. **The schemas were over-eager.** `contractRequiredSchema`
   demanded `payment_schedule` (not every contract has one — some
   bill via separate invoices), `governing_law` (has a sensible
   default), and in `invoiceRequiredSchema`, `invoice_number` (which
   the system could auto-generate) and `due_date` (invoices can be
   net-on-receipt). `requirementsRequiredSchema.deadline_expectations`
   was also a field-name mismatch — the UI wrote to `fields.deadline`.
   Users got blocked on fields that weren't genuinely necessary for
   the recipient to understand the document.

### What changed

| Doc type | Required (v3.5.0) | Required (v3.5.1) |
|---|---|---|
| Proposal | `client_name`, `project_title`, `overview`, `timeline`, `investment_amount` | `client_name`, `project_title`, `overview` |
| Contract | `parties.*`, `project_description`, `payment_schedule`, `governing_law` | `parties.*`, `project_description` |
| SOW | `objectives`, `deliverables[]`, `acceptance_criteria` | `objectives`, `deliverables[]` |
| Requirements | `functional_requirements[]`, `deadline_expectations` | `functional_requirements[]` |
| Invoice | `invoice_number`, `client_name`, `line_items[]`, `due_date`, `total > 0` | `client_name`, `line_items[]`, `total > 0` |
| Delivery | unchanged | unchanged |

Philosophy: validation is a backstop against sending an obviously-
broken document, not a completeness checklist. The user's own
judgement gates optional fields.

### Implementation

- **`types/schemas.ts`** — each `*RequiredSchema` slimmed per the
  table above. Error messages rewritten in user-friendly language
  (no "required", no technical terms).
- **`coercedNumber`** (new, in `types/schemas.ts`) — Zod preprocessor
  that treats numeric strings as numbers, empty/null as missing. Applied
  to `line_items.quantity`, `line_items.rate`, `total`.
  Defense-in-depth against the same class of bug at other boundaries.
- **`formatValidationMessage(issues, action)`** (new, exported from
  `types/schemas.ts`) — takes a Zod error, returns a single sentence:
  > Before sending, please fill: Client name, Overview and 2 more.
  Never includes Zod paths or type names. Collapses duplicates, caps
  visible items at 3. The action word ("sending" vs "finalising")
  tells the user what gesture they were trying to make.
- **`labelForZodPath(path)`** (new, exported) — translates Zod paths
  to human labels using a `FIELD_LABELS` map with a sensible fallback
  (title-cased, spaces instead of underscores). Adding a new required
  field also adds its label here.
- **`validateFinalDoc` in `app/dashboard/actions/documents.ts`** —
  return type changed from `string[] | null` to `string | null`, the
  formatted message. Both call sites (`updateDocument` finalize-path
  and `shareDocument` send-path) updated to surface it directly.
- **`ContractSection.addPayment`** (new, `app/dashboard/personal/paperwork/page.tsx`)
  — replaces the inline `onClick={() => {...}}` that pushed raw state.
  Coerces amount to a number, refuses invalid input. Matches
  `InvoiceEditor.addItem`'s posture.

### Backwards compatibility

Existing documents with stringified `payment_schedule.amount` values
(from before the writer fix) continue to validate after the schema
tightening because:

1. The required schema no longer demands `payment_schedule` at all.
2. If future code reinstates a `payment_schedule` requirement,
   `coercedNumber` handles the string values transparently.

Existing documents that were savable under the v3.5.0 required
schemas remain savable under the v3.5.1 ones (the v3.5.1 set is a
strict subset of v3.5.0).
